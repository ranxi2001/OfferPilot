## 3. 考察维度

这道题主要考察以下能力：

### 1. Agent 产品定位理解

你是否知道主流 Agent，比如 Cloud Code、Codex、Hermes，通常是面向开放任务的通用型 Agent，而你的项目可能是垂直场景 Agent。

### 2. Agent 架构认知

面试官想听的不只是“他们更复杂，我的更简单”，而是：

- 任务规划方式有什么不同？
- 工具调用体系有什么不同？
- 状态管理有什么不同？
- 上下文管理有什么不同？

> 我不是在做通用 Agent，而是在做一个垂直场景下更可控、更稳定、更容易评估和交付的 Agent。

### 4. 项目边界和优势表达

面试官想知道你是否能清楚说出：

- 你的 Agent 不擅长什么？

## 7. 参考回答

下面是一版更适合面试场景的回答：

> 我理解 Cloud Code、Codex、Hermes 这类主流 Agent 更偏通用型 Agent，它们的目标是处理开放域任务，比如代码修改、文件操作、复杂推理、多轮工具调用和跨任务规划。所以它们通常需要更复杂的 planner、tool registry、上下文管理、执行沙箱、状态追踪和失败恢复机制。
>
> 我的 Agent 项目定位不太一样，它不是要做一个开放域通用 Agent，而是面向具体业务场景的垂直任务型 Agent，比如 PPT 生成、简历优化这类高频但边界比较清晰的任务。我的核心思路是主动收敛任务边界，用更固定的工作流和更少但更稳定的工具，换取更高的可控性和交付稳定性。
>
> 具体来说，主流 Agent 的动作空间很大，它可能需要自己判断要不要读文件、写代码、跑测试、调用外部工具，还要在多轮 observe-act 循环中不断修正。而我的 Agent 通常会把流程拆成几个确定阶段。比如 PPT Agent 里，我会先做需求解析，然后生成大纲，再补充资料，接着生成页面结构，最后调用 PPT 生成工具导出结果。每个阶段能调用的工具是有限的，所以状态管理和错误恢复会比通用 Agent 更简单，也更可控。
>
> 工具设计上，Codex 或 Cloud Code 这类 Agent 会有比较丰富的文件、终端、代码执行、检索等工具。而我的项目里工具数量更少，主要围绕业务闭环设计，比如简历解析工具、岗位匹配工具、PPT 模板工具、内容生成工具、格式校验工具等。这样牺牲了一部分泛化能力，但在具体任务上输出更稳定，也更容易做质量评估。
>
> 所以总结一下，二者的差异不是简单的强弱关系，而是定位不同。主流 Agent 更适合开放式复杂任务，我的 Agent 更适合边界清晰、流程可控、需要稳定交付的垂直场景。我的主要工程取舍是用较窄的任务边界和受控工具链，换取更低成本、更低延迟、更好的格式一致性和更稳定的用户体验。
