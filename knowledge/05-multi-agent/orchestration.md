# 多 Agent - 编排模式

## Q：多 Agent 系统有哪些编排模式？各自的适用场景是什么？

> 来源：某 AI 平台团队面试

**新手答**："就是多个 Agent 配合工作，一个管调度，其他干活"

**高手答**：

多 Agent 编排的核心问题是控制流归谁管。主要有四种模式：

1. **Orchestrator 模式（中心化）**：一个主 Agent 分解任务、分发子任务、汇总结果。优点是流程可控、可审计；缺点是主 Agent 是瓶颈和单点故障。适合任务可以清晰分解的场景（如代码审查：分别检查安全、性能、风格）。

2. **Pipeline 模式（串行）**：Agent A 的输出是 Agent B 的输入，线性流转。适合有明确前后依赖的流程（如：需求分析 → 代码生成 → 测试编写 → 代码审查）。

3. **Debate/Consensus 模式（对抗）**：多个 Agent 独立给出方案，通过辩论或投票达成共识。适合需要多角度验证的场景（如事实核查、方案评审）。缺点是 token 消耗大。

4. **Swarm 模式（去中心化）**：Agent 之间直接通信，无中心调度。适合动态变化的环境，但调试困难，生产中很少用。

工程关键点：
- 子 Agent 的 context 隔离：共享太多会超 context window，太少会丢失关键信息
- 错误传播：子 Agent 失败不应该雪崩整个系统，要有 circuit breaker
- Token 预算分配：给每个子 Agent 设 maxTokens，避免一个 Agent 吃光全部 budget
- 可观测性：每个 Agent 的输入输出都要结构化记录，否则多跳问题无法排查

- 系统设计能力
- 对分布式系统模式的理解迁移
- 生产环境的工程意识
