# 多 Agent - 上下文隔离与共享

## Q：多 Agent 系统中，如何处理子 Agent 之间的上下文隔离与共享？

> 来源：某 AI 平台架构面试

**新手答**："把主对话历史传给所有子 Agent"

**高手答**：

上下文管理是多 Agent 系统最容易翻车的地方。传太多——token 爆炸、信息噪声；传太少——子 Agent 缺乏关键背景做出错误判断。

**设计原则**：

1. **Need-to-know 原则**：每个子 Agent 只接收与其任务相关的最小上下文。诊断 Agent 不需要知道 JD 分析的细节，反之亦然。

2. **摘要注入**：不传原始对话历史，而是由主 Agent 或专门的 summarizer 生成任务相关摘要。比如："用户正在准备 XX 公司的 YY 职位面试，目标维度是 Agent 架构"。

3. **结构化上下文**：
```
[任务目标] 诊断这道面试题的回答质量
[用户背景] 3年经验，熟悉 Python，首次接触 Agent 开发
[约束条件] 重点关注工程实践而非理论
```

4. **分层隔离**：
- 全局上下文：所有子 Agent 共享（用户 profile、session 元数据）
- 任务上下文：仅相关子 Agent 可见（当前问题、历史答案）
- 工作上下文：子 Agent 自己产生的中间状态（不回传主循环）

**工程实现**：

- 每个子 Agent 有独立的 messages 数组，不共享引用
- 主 Agent 维护 context budget，按子任务优先级分配
- 子 Agent 返回结果时只返回结论，不返回推理过程（除非 debug 模式）
- 如果子 Agent 需要追加上下文，通过显式的 context_request 机制，不自动灌入

## 考察点

- 系统架构边界设计
- Token 经济学意识
- 信息论基础（最小充分信息）
