# 评测质量 - LLM-as-Judge

## Q：用 LLM 做评测打分有什么坑？怎么提高可靠性？

> 来源：某 AI 评测团队面试

**新手答**："让 GPT-4 打分就行，它很准"

**高手答**：

LLM-as-Judge 是目前最流行的自动化评测方案，但有系统性偏差需要处理：

**已知偏差**：

1. **Position Bias**：倾向于给排在前面的选项更高分。解决：随机化顺序 + 多次评估取平均。
2. **Verbosity Bias**：更长的回答倾向于获得更高分。解决：prompt 中强调"简洁不扣分"。
3. **Self-enhancement Bias**：模型倾向于给自己生成的内容更高分。解决：用不同模型生成和评估。
4. **Sycophancy**：倾向于给积极正面的内容更高分。

**提高可靠性的方法**：

1. **Rubric 设计**：给出明确的评分标准和示例，不要让模型自由发挥。每个分数段要有 anchor example。
2. **多维度评分**：不要一个总分，拆成 correctness、completeness、clarity 分别打分再加权。
3. **Chain-of-Thought**：要求先写评估理由再打分，能减少随机性。
4. **多次采样**：同一对(query, response)评估 3-5 次，取中位数。T=0.3 减少方差。
5. **人机校准**：用人工标注集校准 LLM 评分，计算 Cohen's Kappa 看一致性。

工程架构：
- 评估 pipeline 要和生成 pipeline 解耦，可以异步/批量运行
- 存储结构化的评估结果（维度分、理由、原始输出），便于分析趋势
- 定期用人工抽检 LLM 评分是否漂移
