# Harness 工程 - Agent 错误处理

## Q：Agent 系统中如何设计错误处理和恢复机制？

> 来源：某 AI infra 团队面试

**新手答**："try-catch 捕获异常，返回错误信息给用户"

**高手答**：

Agent 的错误处理比普通应用复杂，因为错误可能发生在 LLM 调用、工具执行、权限校验等多个环节，且需要让 LLM 参与恢复决策。

设计原则：

1. **错误分类**：
   - Transient（网络超时、限流）→ 自动重试，指数退避
   - Input error（参数不合法）→ 结构化错误返回 LLM 让它修正
   - Fatal（API key 无效、服务不可用）→ 终止循环，通知用户
   - Logic error（模型幻觉、工具不存在）→ 错误信息喂回模型

2. **is_error 语义**：Tool result 必须带 is_error 标记。纯字符串错误信息会被 LLM 当成正常输出处理，导致错上加错。

3. **重试策略**：
   - LLM API 层：指数退避 + jitter，区分 429（等待）和 500（换 provider）
   - Tool 执行层：按工具配置决定是否重试，幂等工具可重试
   - Agent 循环层：连续 N 次工具失败后降级（简化任务或直接文本回复）

4. **Circuit Breaker**：某个工具连续失败超过阈值，临时禁用，避免浪费 token 反复尝试。

5. **Graceful Degradation**：
   - 搜索工具不可用 → 基于已有 context 回答，标注"未经搜索验证"
   - 子 Agent 超时 → 主 Agent 接管，用简化方案

## 考察点

- 分布式系统设计思维
- 对 LLM 特殊性的理解（错误可以喂回模型）
- 生产环境可靠性工程
