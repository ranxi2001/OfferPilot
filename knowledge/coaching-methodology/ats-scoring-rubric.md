# ATS 简历评分体系：7 维度加权评分

> 基于对多个开源 ATS 评分项目的研究提炼，编码了招聘系统筛选简历的核心逻辑。

---

## 评分维度与权重

| 维度 | 权重 | 评估内容 | 达标标准 |
|------|------|---------|---------|
| 关键词匹配 | 30% | JD 中的技能/技术词出现率 | ≥70% 覆盖率 |
| 经验相关性 | 15% | 年限+领域匹配 | 年限达标+领域命中 |
| 格式合规 | 15% | ATS 可解析性 | 标准分节、无纯图片 |
| 动作动词 | 10% | 强动词 vs 弱动词比例 | 强动词占比 ≥60% |
| 职位匹配 | 10% | Title 关键词对齐 | 核心职位词出现 |
| 量化表达 | 10% | 数字/指标密度 | 每段经历至少 1 个量化 |
| 可读性 | 10% | 行长/分段/结构 | 适中行长+清晰分节 |

## 等级划分

| 等级 | 分数区间 | 含义 | 建议动作 |
|------|---------|------|---------|
| A | 90-100 | 高匹配，ATS 必过 | 直接投递 |
| B | 80-89 | 良好匹配 | 微调后投递 |
| C | 70-79 | 一般匹配 | 补足关键词后投递 |
| D | 60-69 | 匹配度偏低 | 需针对性重写 |
| F | 0-59 | 高风险被过滤 | 考虑换方向或大幅重写 |

## 关键词权重分级

不是所有关键词权重相同：

| 分类 | 权重倍数 | 示例 |
|------|---------|------|
| 硬技能/技术栈 | 2x | Python, Kubernetes, PyTorch, LangChain |
| 职能/Title 词 | 1.5x | 架构设计, 系统设计, Agent 开发 |
| 业务上下文 | 1x | 电商, 金融, 推荐系统 |
| 软技能 | 0.5x | 沟通, 协作, 自驱 |

**实操建议**: JD 中出现 3 次以上的词，简历中至少出现 1 次。

## 强动词 vs 弱动词

### 强动词（推荐使用）
设计、实现、主导、优化、搭建、推动、构建、开发、部署、交付、解决、重构、迭代、落地

### 弱动词（应替换）
负责、参与、协助、了解、使用、接触、支持、维护

### 替换示例
| 弱 | 强 |
|---|---|
| 负责了系统开发 | 主导设计并实现了系统 |
| 参与了项目 | 独立负责 X 模块的设计与交付 |
| 了解 Docker | 基于 Docker 搭建了生产部署管线 |

## 格式合规检查清单

- [ ] 使用标准分节标题（教育背景、工作经历、项目经历、专业技能）
- [ ] 避免纯表格/图片/多栏布局
- [ ] 联系方式在头部，包含姓名+邮箱+电话
- [ ] 文件格式为 PDF（非 Word/图片）
- [ ] 每段经历有明确的时间区间
- [ ] 无乱码/特殊字符/非标准字体

---

## 常见扣分陷阱

1. **关键词堆砌**: 简历末尾堆一堆不相关技能 → ATS 可能标记为垃圾
2. **缺少时间线**: 经历没有日期 → ATS 无法判断经验年限
3. **文件名不规范**: `简历final(2)(3).pdf` → 体现不专业
4. **一版简历投所有岗位**: 关键词覆盖率必然低 → 被筛掉概率高
5. **过度依赖 AI 生成**: 模板化语言、对称句式 → 越来越多 HR 会标记
